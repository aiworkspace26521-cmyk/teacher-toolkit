## 1. 徽章補償機制的問題

**位置**：`simulation-test.js:1003-1006`、`kpi-core.js:491-494`

```javascript
const totalSubmitDays = state.submitDates ? Object.keys(state.submitDates).length : 0;
if (totalSubmitDays >= 60) state.badges += 5;
else if (totalSubmitDays >= 30) state.badges += 2;
```

**影響追蹤**：
- D30（M1 末）：MAX submitDays=30 → +2，徽章 4→6
- D35（M2W1）：`gymIndex=min(6,31)=5` → 娜姿(badge6)
- 但 `badgeGain=(gym.badge===badges+1?1:0)=(6===7?1:0)=0` → **卡住無法取得下一徽章**
- 因為 `gymIndex` 永遠指向 badge6，但條件檢查過不了

**對 MAX 的影響**：
- 每次 recalculateState 都 +2（D30~D59），徽章數虛胖
- D60 時 +5 取代 +2，徽章再跳一次
- 最終 32 徽章不變（因 D224 之後還夠時間補），但節奏被打亂

**對 MIN 的影響**：
- MIN 原本實力只能拿 16 徽章，補償灌到 21（+5）
- 造成「徽章數遠高於實力」的假象

**結論**：完全同意取消。這兩段可以直接刪除。

## 3.2 由骨架拓展：館主/小兵/四天王/路人等級設定

### 核心公式（維持不變）

來自 `generateEnemy` 與 `generateGymWaves`：

- **道館館主 wave**：`baseLevel = max(5, playerLv + gym.lvBonus)`
- **道館小兵 wave**：`baseLevel = max(5, playerLv + floor(gym.lvBonus × 0.4))`
- **野生路人戰**：`baseLevel = max(5, playerLv + tierBonus + regionAdj)`
- **四天王**：`baseLevel = max(5, playerLv + e4.lvBonus)`
- **冠軍**：`baseLevel = max(5, playerLv + champ.lvBonus)`

### 核心結論

1. **徽章補償完全取消** — 直接刪除兩處 if/else 即可
2. **舊 M1W5~M2W5 → M2W1~M3W2** — 命名位移 1~2 週
3. **館主等級曲線**：lvBonus 從 0→88，張力集中在 M5W4 之後（合眾中後期跳升至 lvBonus 54+）
4. **四天王可挑戰時間**：關都 E4 (M3W1) 最早，卡洛斯 E4 (M7W4) 最晚
5. **MAX 與 MIN 的等級差**：M1 差 3 級、M7 差 47 級 → 道館設計須涵蓋兩種玩家的體驗
