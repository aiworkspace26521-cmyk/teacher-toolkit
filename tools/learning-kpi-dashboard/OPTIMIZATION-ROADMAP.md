### U2：戰鬥 EXP 前後端漂移修復

#### 現狀問題
`frontend/kpi-dashboard.html` 中 `finishBattle`（line 3430-3465）計算 EXP 時：
```
for (var j = 0; j < partyUsed.length; j++) {
  share = j === 0 ? baseExp×1.2 : baseExp×0.3;
  totalExpGain += share;
}
// 僅 sum 參戰者 EXP，非參戰者不計入
```

但 `recalculateStudentState`（line 884-889）重算時認為：
```
非參戰隊員 → 獲得 expGained 的 50%
學習裝置持有者 → 獲得 100% expGained
```

**結果**：事件重算與實際遊玩 EXP 分配不一致，造成 Firestore 狀態漂移。

#### 修復方案
在 `finishBattle` 的 EXP 計算中加入非參戰隊員的留守 EXP，使 `totalExpGain` 正確反映全隊分配：

```
totalExpGain = sum(參戰者) + sum(非參戰者×0.5) + sum(學習裝置持有者×0.5補償)
```

#### 影響範圍
| 系統 | 影響 | 說明 |
|------|:----:|------|
| 戰鬥系統 | 🔴 直接 | EXP 分配邏輯改變 |
| 事件溯源 | 🔴 直接 | 歷史事件重算結果不受影響（recalculateStudentState 不變） |
| EXP 曲線 | 🟡 中 | 新戰鬥 EXP 總量增加（約 +30~50%） |
| 9 月模擬預測 | 🟡 中 | 需重新跑 simulation-test.js 確認偏移量 |
| 所有已存在的事件 | 🟢 無影響 | 舊事件不重新寫入 |

#### 風險評估
| 風險 | 等級 | 緩解措施 |
|------|:----:|---------|
| EXP 總量暴增導致 M2 就 Lv.99 | 🔴 | 修復後 simulation-test 需確認 MAX 仍在 M6 後達 Lv.99 |
| 參與倍率需同步調降補償 | 🟡 | 可搭配 U3 一起做（參與×1.2/0.5） |
| 舊事件 replay 與新邏輯不一致 | 🟢 | recalculateStudentState 不變，僅新事件受影響 |

#### 測試驗證
1. **單元測試**：寫入假資料確認非參戰隊員 EXP 正確計入
2. **模擬測試**：simulation-test.js 跑 9 月模擬，確認 MAX/MIN 曲線合理
3. **回歸測試**：18 項 Playwright E2E 測試全部通過
4. **漂移檢查**：新戰鬥後 recalculateStudentState 與實際 totalExp 差異 <1%

### U5：PvP 重製 P0 — 即時邀請 + 3vs3 隊伍戰

#### 設計目標
1. 兩個小朋友可以**即時互邀**對戰
2. 使用**完整的 3vs3 隊伍**（從 Party 選 3 隻），而非目前 1vs1
3. 戰中可手動切換寶可夢

#### 技術架構
```
pvp_battles/{battleId}          ← 對戰文件
  ├─ challengerId: "Neil"
  ├─ defenderId: "Emma"
  ├─ status: "invited"|"accepted"|"in_progress"|"completed"
  ├─ partyChallenger: [pkmn1, pkmn2, pkmn3]
  ├─ partyDefender: [pkmn1, pkmn2, pkmn3]
  ├─ activeChallenger: 0        ← 當前上場 index
  ├─ activeDefender: 0
  ├─ turn: 1
  ├─ currentTurn: "challenger"
  ├─ log: []
  └─ result: null|"challenger_win"|"defender_win"
```

#### 流程
```
挑戰者點「PvP」→ 選擇對手（Neil/Emma/管理員）→ 選擇 3 隻隊伍
  → Firestore 寫入 pvp_battles（status=invited）
  → 被挑戰者 onSnapshot 監聽到新邀請 → 顯示通知氣球
  → 接受 → status=accepted → 雙方進入戰鬥畫面
  → 回合制：各自選招 → 比速度決定先攻順序
  → 昏厥時可切換下一隻
  → 全員昏厥 → 分出勝負
```

#### 現有程式碼可用資源
| 資源 | 位置 | 用途 |
|------|------|------|
| 現有戰鬥引擎（startBattle/executeTurn） | line 3300-3600 | 傷害計算、天氣、特性 |
| 現有 PvP 回合邏輯（executePvPRound） | line 4410-4428 | 速度判定模板 |
| 現有 ELO 排名系統 | line 4450-4490 | 可沿用 |
| Firestore onSnapshot | line 700-720 | 即時同步基礎設施已存在 |

#### 影響範圍
| 系統 | 影響 | 說明 |
|------|:----:|------|
| PvP 系統 | 🔴 全面改寫 | 新資料結構 + 新流程 |
| Firestore 規則 | 🟡 需新增 | pvp_battles 集合讀寫規則 |
| 戰鬥引擎 | 🟡 共用 | 現有 executeTurn 可大部分重用 |
| 成就系統 | 🟢 相容 | 現有 PvP 成就條件可沿用 |
| ELO 排名 | 🟢 相容 | 現有 pvp_ranking 集合可沿用 |
| 現有 PvP 資料 | 🟢 向下相容 | 舊 pvp_ranking 資料保留 |

#### 風險評估
| 風險 | 等級 | 緩解措施 |
|------|:----:|---------|
| Firestore 即時同步延遲 | 🟡 | onSnapshot 通常 <500ms，可接受 |
| 雙方同時操作衝突 | 🔴 | 以 Firestore transaction 確保回合原子性 |
| 戰鬥引擎共用函數修改影響道館/路人戰 | 🔴 | PvP 戰鬥使用獨立副本或加參數隔離 |
| 小朋友不會操作 | 🟡 | UI 引導（閃爍按鈕 + 提示文字） |

#### 測試驗證
1. **邀請流程測試**：寫入 pvp_battles → 確認 onSnapshot 觸發
2. **隊伍選擇 UI**：可選 1~3 隻後 confirm，下限 1 上限 3
3. **戰鬥流程測試**：完整對戰（含切換）至分出勝負
4. **衝突測試**：雙方同時送招 → 確認速度判定正確
5. **斷線恢復**：一方重整頁面 → 戰鬥狀態應可恢復
6. **所有現有 Playwright 測試通過**
7. **手動操作測試**：真人實測 Neil vs Emma

## 5. 測試策略

```

  ↓

│

     │
```

## 7. 部署回滾計畫

### 7.1 標準部署流程

```
修改 frontend/kpi-dashboard.html
  ↓
同步至 public/kpi-dashboard.html (手動或 sync-public.ps1)
  ↓
執行 simulation-test.js ✅
  ↓
執行 Playwright E2E ✅
  ↓
git commit + git push
  ↓
Firebase 自動部署 (GitHub Actions)
```

### 7.2 回滾條件

符合以下任一條件即觸發回滾：

1. **模擬測試 FAIL** → `MAX Lv.99 提早超過 1 個月` 或 `MIN 徽章數倒退`
2. **Playwright FAIL ≥3 項**（排除已知 flaky 測試）
3. **親密度顯示全白/亂碼/NaN**
4. **PvP 雙方不同步**（一方看到結果另一方卡住）
5. **Firestore 寫入失敗**（戰鬥/PvP 事件未寫入）

### 7.3 回滾步驟

```bash
# 步驟 1：還原程式碼
git revert HEAD --no-edit
git push

# 步驟 2：重新部署（CI 自動）
# 或手動：firebase deploy --only hosting

# 步驟 3：驗證回滾成功
# 確認 https://opencodefirebase.web.app/kpi 恢復上一個正常版本

# 步驟 4：記錄問題
# 更新 OPTIMIZATION-ROADMAP.md 踩坑筆記
```

### 7.4 部分回滾（單一功能）

若僅單一功能（如 PvP）出問題，可使用 git revert 特定 commit 而非全部：

```bash
# 找出 PvP 相關 commit
git log --oneline --all | Select-String "PvP"

# 還原特定 commit
git revert <commit-hash> --no-edit
```
