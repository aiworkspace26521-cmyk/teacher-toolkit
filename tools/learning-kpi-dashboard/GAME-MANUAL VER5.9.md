## 1. 遊戲概述

學習 KPI 管理系統將 Pokémon 遊戲化架構重建為學習管理系統，
保留 gamification 核心（等級、經驗值、成就、收集、對戰）但以「學習任務驅動」取代「戰鬥模擬」。

### 1.1 學生角色

| 角色 | 說明 |
|------|------|
| **Neil（苡逢）** | 學生 A，五年級 |
| **Emma（苡瑄）** | 學生 B，五年級 |
| **管理員** | 家長/教師，可模擬時間、發放資源、重置進度 |

### 1.2 核心資源

| 資源 | 說明 | 獲取方式 |
|------|------|---------|
| **EXP（經驗值）** | 提升寶可夢等級 | 每日任務提交、戰鬥勝利、商城道具 |
| **金幣** | 商城貨幣 | 每日任務提交（門檻制）、戰鬥勝利 |
| **徽章** | 道館挑戰成就 | 道館戰勝利（冷卻 ≥1 天） |
| **道具** | 戰鬥消耗品/永久強化/進化 | 商城購買 |
| **親密度** | 影響特定進化條件 | 每日提交、捕捉、戰鬥勝利、橙橙果餵食 |

## 3. 每日任務與分數系統

### 3.1 任務類別

| 類別 | 科目 | 滿分 | 任務數 |
|------|------|:----:|:-----:|
| 數理道館 | 代數、計算 | 30 | 2 |
| 國語道館 | 國學、閱讀 | 25 | 2 |
| 英文道館 | PET/FCE、閱讀 | 25 | 2 |
| 紀律與體能 | 運動、小提琴、家務、睡眠 | 20 | 4 |
| **每日滿分** | | **100** | **10** |

### 3.2 分數計算

```
最終分數 = 勾選任務分數總和（無倍率）

健康違規制約（Veto 系統）：
- 若「運動」與「就寢」未同時勾選且分數 ≥50 → 顯示 ⚠️ 警告
- 若「運動」與「就寢」未同時勾選且分數 ≥80 → 強制上限 79 分
- 健康違規時金幣 = 0 幣
```

### 3.3 提交模式

| 模式 | 選擇方式 | EXP 公式 | 金幣公式（門檻制） | 效果 |
|------|---------|----------|-----------------|------|
| 一般提交 | 預設 | floor(分數 × 10 × max(1, 基準等級/5)) | 見下方門檻表 | 全隊均衡成長 |
| 強化現有技能 | 選「強化」+ 指定寶可夢 | floor(分數 × 10 × max(1, 基準等級/5)) × rubberband | 同左 | 集中灌注指定寶可夢 |
| 挑戰新目標 | 選「捕捉」+ 選擇屬性 | **0 EXP**（僅獲取寶可夢） | 同左 | 獲得新寶可夢（依分數門檻） |

**金幣門檻制（門檻級距，非線性）：**
| 門檻 | 金幣 |
|:----:|:----:|
| 分數 = 100 | 50 幣 |
| 80 ≤ 分數 < 100 | 30 幣 |
| 50 ≤ 分數 < 80 | 10 幣 |
| 分數 < 50 | 0 幣 |
| 健康違規（運動+就寢未同時勾選） | 0 幣 |
| 護符金幣（持有） | ×1.5（無條件進位） |

**Rubberband EXP 機制（強化模式專屬）：**
| 目標等級 vs 基準等級 | 倍率 |
|:-------------------:|:----:|
| 目標等級 < 基準等級 | ×1.5 🚀 追趕紅利 |
| 基準等級 ≤ 目標 ≤ 基準+5 | ×1.0 正常 |
| 目標 > 基準+5 | ×0.3 ⚠️ 等級壓制 |

### 3.4 時間系統

- 月份計算：以 2026 年 4 月為基準（第 1 月），自動遞增
- 管理員可透過下拉選單模擬月份（1~10）與週狀態（一般道館/聯盟大會）
- 聯盟時期僅影響聯盟戰開放（不影響每日分數）

### 3.5 每日提交防重複（V5.5）

每日提交在送出前進行 **Firestore 雙重檢查**：
1. 查詢今日 `kpi_events` 是否有 `action === "每日提交"`
2. 若有已存在的提交記錄 → 顯示 Toast 警告並 `return`，阻止重複提交
3. 時間上界 `< tomorrowStart` 確保跨日計算準確
4. 此機制對非 Admin 用戶生效；Admin 跳過此檢查（設計）

## 4. 捕捉系統

### 4.1 分數門檻與稀有度

捕捉系統有 **徽章門檻限制**：需累積足夠徽章才能遇到傳說寶可夢。

| 每日分數 | 徽章 <8 | 徽章 8~15 | 徽章 ≥16 | 捕捉等級 |
|---------|---------|----------|---------|---------|
| 0~49 | ❌ 無法捕捉 | 同左 | 同左 | — |
| 50~59 | 100% 一般 | 同左 | 同左 | Lv.6~7 (score/8) |
| 60~74 | 85% 一般 / 15% 稀有 | 同左 | 同左 | Lv.7~9 (score/8) |
| 75~94 | 65% 一般 / 35% 稀有 | 61.75% 一般 / 33.25% 稀有 / **5% 傳說** | 同左 | Lv.12~15 (score/6) |
| 95~100 | 40% 一般 / 60% 稀有 | 37.6% 一般 / 56.4% 稀有 / **6% 傳說** | 同左 | Lv.23~25 (score/4) |

### 4.2 屬性選擇
開啟「挑戰新目標」時顯示屬性下拉選單，支援 18 屬性 + 隨機。

### 4.3 捕捉流程
1. 勾選任務、選擇「挑戰新目標」
2. 選擇指定屬性（或隨機）
3. 預覽顯示可達到的稀有度等級
4. 提交後系統根據分數判定稀有度 + 從屬性池隨機選取
5. 生成寶可夢 ID、等級、雙屬性
6. 事件寫入 Firestore（action: \捕捉\）
7. Toast 通知捕捉結果

### 4.4 每日保底捕捉（P5-3b）
| 連續提交天數 | 保底獎勵 |
|:-----------:|---------|
| 5 天 | 手動選擇一隻**稀有級**寶可夢 |
| 10 天 | 50% 傳說 / 50% 稀有（隨機抽選） |
- 使用 Firestore 追蹤連續提交天數（submitStreak）
- 儀表板顯示火焰動畫 + 預覽保底提示
- 需每日確實提交任務才計入連續天數

### 4.5 MIN 補償（P7-3）
長期提交但少戰鬥的玩家自動獲得徽章補償（在 recalculateStudentState 中計算）：
| 總提交天數 | 自動獲得徽章 |
|:---------:|:-----------:|
| ≥30 天 | +2 徽章 |
| ≥60 天 | +5 徽章 |

## 8. 進化系統

### 8.1 四種進化方式

| 方式 | 條件 | 數量 |
|------|------|:----:|
| 等級進化 | EVO_STAGE_MAP 決定階段數（1~3），門檻 = stage×15 | 多數 |
| 親密度進化 | 親密度 ≥120 + Lv.20+ | 2 種 |
| 道具進化 | Lv.30+ + 持有對應道具 | 8 種 |
| 通訊交換進化 | 透過交換系統傳送 | 4 種 |

### 8.2 等級進化

每種寶可夢依 `EVO_STAGE_MAP[speciesName]`（1~3 階段）決定進化門檻：

```
門檻 = EVO_STAGE_MAP[speciesName] × 15
stage = Math.floor(Math.min(level, 99) / 15)
- stage === 0 → 基礎型態（回傳 species.name，不使用 evolutions[0]）
- stage >= 1  → 進化型態（evolutions[stage - 1]）
- 最多進化至 EVO_STAGE_MAP 指定的階段數
```

V5.6 修正 `getEvolvedName` index 偏移（2026-06-08）：
- 原 bug：`evolutions[Math.floor(level/15)]`，index 0 回傳 `evolutions[0]`（進化型態）而非 `species.name`（基礎型態）
- 導致所有 LV0-14 寶可夢被顯示為進化型態
- 修復：`stage===0` 時回傳 `species.name`，只有 `stage>=1` 才用 `evolutions[stage-1]`

`checkEvoReady` 與 `doEvolve` 使用 `EVO_STAGE_MAP[speciesName] * 15` 計算進化門檻，
確保每種寶可夢依其進化鏈階段數（1~3 階段）正確對應進化等級。

## 9. 戰鬥系統

### 9.1 戰鬥流程
```
回合制輪流攻擊（速度高者先攻）
  ├─ 天氣效果（回合開始判定）
  ├─ 特性觸發（出場/回合開始）
  ├─ 狀態檢查（凍結/睡眠/麻痺）
  ├─ 招式選擇（4 選 1）
  ├─ 傷害計算（屬性相剋 + 能力階級 + 道具）
  ├─ 狀態/異常判定
  ├─ 持續傷害（燒傷/中毒/寄生種子）
  └─ 切換下一隻（若當前昏厥）
```

### 9.2 傷害公式
```
物理：baseDamage = floor(((Lv×2/5+2)×power×atkStat/defStat)/50+2)
特殊：baseDamage = floor(((Lv×2/5+2)×power×spatkStat/spdefStat)/50+2)

修正（依序疊乘）：
1. 天氣（雨/晴/沙/雹）
2. 特性（猛火/激流等）
3. 適應力 STAB ×1.33
4. 道具（達人帶×1.2、生命寶珠×1.3、AV背心×1.5）
5. 傳說互打 ×1.5
6. 屬性相剋 ×0/0.5/1/2
7. 最小值 max(1, dmg)
8. 隨機 ×0.85~1.0
```

## 21. 資料架構

```
events/{eventId}             → 事件溯源（不可變日誌）
students/{studentId}         → 學生當前狀態（由事件重算）
subjects/{subjectId}         → 科目定義（任務模板）
trade_offers/{offerId}       → 交換請求（暫存）
```

所有操作採用**事件溯源（Event Sourcing）**模式：
- 資料不可變：只新增、不修改/刪除
- 狀態由事件重算（\recalculateStudentState\）
- 前端使用 Firestore 即時同步 + sessionStorage 快取
- 批量寫入合併（500ms debounce）

### 21.1 安全性強化
- Firestore rules 確保 trade_offers 僅學生可操作自己的資料
- `kpi_events`：僅可新增+讀取，不可修改/刪除
- 學生資料去識別化（僅用座號）

## 26. 開發者指引

### 本地開發

```bash
# 1. 模擬測試（無需 Firebase）
node tools/learning-kpi-dashboard/test/simulation-test.js

# 2. E2E 測試（需部署至 Firebase 或本地伺服器）
npm run test:e2e

# 3. 快速模擬測試
npm run test:sim
```

### 部署流程

1. 修改 `frontend/kpi-dashboard.html`
2. 同步修改 `public/kpi-dashboard.html`（Firebase Hosting 部署版本）
3. 執行模擬測試確認無回歸
4. 提交 PR → CI 自動執行測試
5. 合併至 master → 自動部署至 GitHub Pages + Firebase Hosting

### 重要約定

- **雙檔案同步**：`tools/learning-kpi-dashboard/frontend/kpi-dashboard.html` 與 `public/kpi-dashboard.html` 必須保持一致
- **模擬測試獨立常數**：`simulation-test.js` 有自己的常數副本（MOVE_DATABASE、GYM_LEADERS 等），前端修改後需同步更新
- **事件溯源**：學生狀態由 `kpi_events` 日誌重新計算，不可直接修改 `kpi_students` 欄位
- **去識別化**：學生資料使用座號，不使用真實姓名

| Firestore 規則/索引檢查 | 每次 push | ~5s |
